# AI Backend Intelligence Spec

This thread owns:

- AI response contract
- WhatsApp routing and access behavior
- Document generation pipeline
- Factuality/refusal behavior
- Audit and denial logging
- Missing-data and conflict handling
- Substitution ranking logic

This spec must honor locked UX/product rules:

- Default replies are short structured bullets
- Only label AP classes; if not AP, show no program label at all
- Cross-department substitution is blocked by default unless explicitly overridden
- HoD cannot see names outside department unless explicit override exists
- Teachers can only be reported as: class, meeting, free, or off-campus
- Duties are shown only for non-teaching staff

## 1. AI Response Contract

The backend should stop relying on loosely enforced JSON from the LLM. Use a strict response envelope with two output modes:

- `plain_text`
  Used when the reply is deterministic and generated by code, templates, or safe fallbacks.
- `structured_json`
  Used when the backend needs machine-readable actions, denial reasons, document specs, or audit metadata.

Rule:

- Internal AI calls should always target `structured_json`
- End-user WhatsApp replies should usually render as plain text bullets from structured backend objects
- Raw model prose should never be sent directly unless JSON parsing has already failed and a safe fallback wrapper is applied

### 1.1 Canonical Envelope

```json
{
  "version": "1.0",
  "mode": "structured_json",
  "request_id": "uuid",
  "timestamp": "2026-04-12T18:30:00.000Z",
  "channel": "whatsapp",
  "actor": {
    "lid": "9617xxxxxxx@c.us",
    "role_scope": "teacher",
    "person_id": 123,
    "department_ids": [2],
    "department_names": ["Secondary"]
  },
  "routing": {
    "intent": "schedule_lookup",
    "confidence": 0.96,
    "requires_clarification": false,
    "requires_override": false,
    "override_type": null
  },
  "decision": {
    "status": "allowed",
    "reason_code": "OK",
    "reason_text": "Requester can view own schedule.",
    "policy_refs": ["OWN_SCHEDULE_ALLOWED"]
  },
  "answer": {
    "type": "schedule_status",
    "audience_mode": "user_visible",
    "render_hint": "short_bullets",
    "plain_text": null,
    "blocks": []
  },
  "audit": {
    "source_mode": "deterministic_lookup",
    "sources": [],
    "denied_fields": [],
    "warnings": [],
    "escalation": {
      "needed": false,
      "target": null,
      "reason": null
    }
  },
  "errors": []
}
```

### 1.2 `answer.blocks` Types

Allowed block types:

- `bullets`
- `clarification`
- `denial`
- `document_ready`
- `document_request_missing_fields`
- `substitution_options`
- `handoff_notice`
- `error`

Example:

```json
{
  "type": "bullets",
  "title": "Your Monday Schedule",
  "items": [
    "P1: Grade 8 Math",
    "P2: Free",
    "P3: Meeting"
  ]
}
```

### 1.3 When to Use Structured JSON vs Plain Text

Use `structured_json` when:

- the router is classifying intent
- access control is being evaluated
- document generation is requested
- substitution ranking is requested
- an answer must cite denial reasons, missing fields, warnings, or escalation
- data conflicts or missing schedule records are present

Use `plain_text` only after backend rendering when:

- the result is already validated
- there is no hidden machine action
- the message is a final WhatsApp sendable string

### 1.4 Allowed Decision Status Values

- `allowed`
- `clarify`
- `denied`
- `escalate`
- `partial`
- `error`

### 1.5 Core Reason Codes

- `OK`
- `MISSING_REQUIRED_FIELDS`
- `ACCESS_OWN_ONLY`
- `ACCESS_DEPARTMENT_ONLY`
- `ACCESS_OVERRIDE_REQUIRED`
- `CROSS_DEPARTMENT_BLOCKED`
- `NAMES_HIDDEN_BY_POLICY`
- `DATA_UNAVAILABLE`
- `DATA_CONFLICT`
- `LOW_CONFIDENCE`
- `GENERATION_VALIDATION_FAILED`
- `ESCALATED_TO_OWNER`

### 1.6 Render Rules for WhatsApp

Final user-visible rendering defaults to short structured bullets:

- first line is a short title only when helpful
- then 1 to 4 bullets
- no paragraphs unless the message is a refusal or escalation notice
- no internal codes shown to end users

Examples:

```text
Your Tuesday Schedule
- P1: Grade 9 Biology
- P2: Free
- P3: Meeting
```

```text
Access limited
- You can view staff in your department only.
- This request needs an override to continue.
```

## 2. WhatsApp Message Flow Rules

Backend flow:

1. Ingress normalization
2. Identity resolution
3. Role and department scope load
4. Intent routing
5. Policy gate
6. Deterministic lookup if possible
7. AI generation only if needed
8. Validation and rendering
9. Audit log write
10. Send reply or escalate

### 2.1 Ingress Normalization

Normalize each incoming message into:

```json
{
  "request_id": "uuid",
  "from": "whatsapp id",
  "channel": "whatsapp",
  "message_type": "text|image|document|command",
  "body_raw": "original body",
  "body_normalized": "trimmed normalized text",
  "attachments": [],
  "timestamp": "iso"
}
```

Normalization rules:

- keep original body for audit
- normalize whitespace and casing for routing
- preserve image/document metadata separately
- extract explicit override flags from admin syntax only, never infer override from casual text

### 2.2 Role-Scope Routing

Use these role scopes:

- `teacher`
- `hod`
- `coordinator`
- `office`
- `owner`
- `non_teaching_staff`
- `student`

### 2.3 Access Rules by Query Type

Own schedule:

- teacher/hod/coordinator/office/owner allowed

Other staff schedule:

- teacher denied
- hod allowed only inside same department
- office and owner allowed

Other staff names:

- hod outside department denied by default
- response must hide names rather than partially leaking identity
- office and owner may see names

Substitution planning:

- hod allowed only for own department by default
- cross-department candidates blocked unless explicit override
- office and owner may request override-enabled search

Availability display:

- teaching staff statuses must collapse to:
  - `class`
  - `meeting`
  - `free`
  - `off-campus`
- never expose `on_duty` for teaching staff
- `on_duty` may be used only for non-teaching staff

### 2.4 Override Rules

Override must be explicit and attributable.

Allowed override types:

- `cross_department_substitution`
- `hod_out_of_department_visibility`
- `restricted_identity_visibility`

Override record must include:

- requester id
- approver id
- scope
- target entity or query
- expiry
- reason

Without override:

- do not silently broaden search
- do not reveal blocked names
- do not reveal blocked departments by implication

### 2.5 WhatsApp Response Templates

Allowed:

```text
Access limited
- You can see staff in your department only.
- Ask the office for an override if needed.
```

```text
Need one more detail
- Topic: missing
- Grade: missing
- Type: test, quiz, or worksheet
```

```text
Substitute Options
- P2 Grade 8 Math: Ms X, Mr Y
- P4 AP Grade 11 Chemistry: No in-department free teacher found
```

Escalation:

```text
Escalated
- I could not safely complete this request.
- The office or owner has been notified.
```

### 2.6 Escalation Triggers

Escalate when:

- override is required but unavailable
- data conflicts make the result unsafe
- identity cannot be resolved for an access-sensitive query
- the model returns invalid structured output twice
- a restricted request targets credentials, private personal data, or hidden names

## 3. Document Generation Pipeline

Goal:

- reliable parsing
- deterministic validation
- explicit missing-field collection
- no best-effort raw prose generation sent directly to the document renderer

### 3.1 Pipeline Stages

1. Request classification
2. Request extraction
3. Missing-field check
4. Policy check
5. Generation brief build
6. LLM JSON generation
7. Schema validation
8. Post-processing normalization
9. Document render
10. Delivery + audit

### 3.2 Supported Request Types

- `test`
- `exam`
- `quiz`
- `worksheet`
- `handout`

Optional later:

- `lesson_plan`
- `letter`

### 3.3 Required Inputs

Canonical request object:

```json
{
  "request_type": "quiz",
  "subject": "Biology",
  "topic": "Photosynthesis",
  "grade_label": "Grade 8",
  "program_label": null,
  "language": "en",
  "teacher_name": "Ms Example",
  "department": "Intermediate",
  "difficulty": "medium",
  "answer_key_requested": true,
  "special_instructions": [],
  "source_constraints": {
    "curriculum": null,
    "must_use_uploaded_text": false
  }
}
```

Required before generation:

- `request_type`
- `topic`
- `grade_label`

Preferred:

- `subject`
- `language`
- `answer_key_requested`

### 3.4 Missing-Field Strategy

If required fields are missing, return `decision.status = "clarify"` with:

```json
{
  "answer": {
    "type": "document_request_missing_fields",
    "blocks": [
      {
        "type": "clarification",
        "title": "Need one more detail",
        "missing_fields": ["topic", "grade_label", "request_type"]
      }
    ]
  }
}
```

Do not ask open-ended prose questions when a structured missing field list is possible.

### 3.5 LLM Output Contract for Document Generation

The document model must return only this shape:

```json
{
  "document_spec": {
    "doc_category": "quiz",
    "title": "Photosynthesis Quiz",
    "subtitle": "Grade 8 | 20 minutes",
    "language": "en",
    "sections": [
      {
        "title": "Section A",
        "items": [
          {
            "kind": "mcq",
            "prompt": "Which gas do plants absorb during photosynthesis?",
            "choices": ["Oxygen", "Carbon dioxide", "Nitrogen", "Hydrogen"],
            "answer": "Carbon dioxide",
            "marks": 1
          }
        ]
      }
    ]
  },
  "answer_key_spec": {
    "sections": []
  },
  "quality_notes": [],
  "warnings": []
}
```

### 3.6 Validation Rules Before Rendering

Reject and retry generation if:

- JSON is invalid
- unsupported `doc_category`
- any item missing `kind`
- `mcq` has fewer than 4 choices
- total marks missing for assessments that require marks
- answer key requested but missing answers
- section array empty

On second failure:

- return safe user error
- log `GENERATION_VALIDATION_FAILED`
- optionally escalate if requested by policy

### 3.7 Renderer Input Model

Do not render from freeform text. Normalize LLM output into a typed internal model:

```json
{
  "kind": "assessment_document",
  "doc_category": "quiz",
  "meta": {
    "title": "Photosynthesis Quiz",
    "subtitle": "Grade 8 | 20 minutes",
    "teacher_name": "Ms Example"
  },
  "sections": [
    {
      "title": "Section A",
      "items": [
        {
          "kind": "mcq",
          "prompt": "Which gas do plants absorb during photosynthesis?",
          "choices": ["Oxygen", "Carbon dioxide", "Nitrogen", "Hydrogen"],
          "answer": "Carbon dioxide",
          "marks": 1
        }
      ]
    }
  ]
}
```

This internal model should be the only accepted input to `document-generator.js`.

### 3.8 Output Artifacts

Generation result envelope:

```json
{
  "status": "success",
  "document_id": "doc_123",
  "files": [
    {
      "role": "document",
      "filename": "photosynthesis_quiz.docx",
      "path": "/generated_docs/photosynthesis_quiz.docx"
    },
    {
      "role": "answer_key",
      "filename": "photosynthesis_quiz_key.docx",
      "path": "/generated_docs/photosynthesis_quiz_key.docx"
    }
  ],
  "render_summary": {
    "section_count": 2,
    "item_count": 12
  },
  "warnings": []
}
```

### 3.9 WhatsApp Delivery for Documents

Success message:

```text
Document ready
- Quiz: Photosynthesis Quiz
- Grade: Grade 8
- Answer key: included
```

Failure message:

```text
Could not generate document
- The request was understood, but the document format failed validation.
- Please try again or ask for a simpler quiz.
```

## 4. Factuality and Refusal Rules

### 4.1 Deterministic First

For schedule, presence, substitution, Wi-Fi, calendar, and identity-scoped access:

- database lookup first
- no LLM hallucinated facts
- LLM may summarize deterministic results, not invent them
- duties never appear for teaching staff
- only non-teaching roles can show duty states

### 4.2 Model May Generate Only

- wording
- formatted user-visible bullets
- documents requested by staff
- clarification questions

### 4.3 Refuse or Deny When

- access scope fails
- hidden names would leak outside policy
- credentials or sensitive private data are requested without authorization
- schedule data is missing or conflicting and unsafe to infer
- cross-department substitution is requested without override

### 4.4 Refusal Style

Refusals must be brief and policy-based:

```text
Access limited
- I can't share that from your current access scope.
- This request needs an approved override.
```

## 5. Audit Logging Rules

Every handled message must write one structured audit row.

### 5.1 Minimum Audit Shape

```json
{
  "request_id": "uuid",
  "timestamp": "iso",
  "actor_lid": "whatsapp id",
  "actor_role_scope": "hod",
  "intent": "substitution_lookup",
  "decision_status": "denied",
  "reason_code": "CROSS_DEPARTMENT_BLOCKED",
  "policy_refs": ["SUBSTITUTION_IN_DEPARTMENT_ONLY"],
  "source_mode": "deterministic_lookup",
  "source_records": {
    "people": [12, 45],
    "schedule_rows": 4
  },
  "user_visible_summary": "Cross-department substitution requires override.",
  "escalated": false,
  "warnings": []
}
```

### 5.2 Denial Logging

Always log:

- why the answer was denied
- which policy blocked it
- whether names were hidden
- whether an override existed and was rejected or absent

### 5.3 Partial Answer Logging

If data is incomplete but a safe subset is returned, log:

- returned fields
- hidden fields
- conflict flags

## 6. Missing Data and Conflict Handling

### 6.1 Safe Partial Responses

Allowed partial examples:

- "On campus today, detailed timetable not mapped"
- "No in-department free teacher found for P3"

Not allowed:

- inferring a teacher is free when timetable rows are missing
- guessing department when person has mixed departments and no scoped match

### 6.2 Conflict Types

- schedule says class but campus map says off-campus
- multiple active department mappings with no primary
- person match ambiguous
- answer key requested but generated spec missing answers

### 6.3 Conflict Behavior

- mark `decision.status = "partial"` or `error`
- include warning codes in audit
- prefer denial or escalation over a misleading answer

## 7. Substitution Ranking Logic Spec

Ranking applies only after policy scope is satisfied.

### 7.1 Eligibility Filters

Candidate must:

- be active
- be substitution eligible
- be on campus in the target period
- be `free` in the target period
- belong to the same department/substitution pool by default

Blocked by default:

- cross-department candidate
- `meeting`
- `class`
- `off-campus`
- ambiguous timetable

### 7.2 Ranking Order

Rank candidates in this exact order:

1. Same department + free + on campus
2. Lowest teaching load that day
3. Closest to absent teacher's subject, if available
4. Show top 3, then "more available"

### 7.3 Override Path

If explicit override is present:

- expand search to approved outside departments
- tag every outside-department candidate with `override_used = true`
- surface that fact in audit, not necessarily in user-facing copy

## 8. Implementation Notes for Current Codebase

Current repo alignment:

- `core/handler.js` should produce and consume the canonical response envelope
- `core/lookup-engine.js` should remain the first path for deterministic operations
- `core/document-generator.js` should accept only validated normalized document models
- `core/worksheet-generator.js` should be retired from freeform-text parsing for staff docs, or used only as a compatibility adapter
- `core/logger.js` should be upgraded from loose message logs to structured audit events

Recommended immediate refactor order:

1. Introduce shared schemas in `core/schemas/`
2. Add a response renderer that turns structured blocks into WhatsApp-safe bullets
3. Split routing, policy, and execution into separate modules
4. Add document-spec validation before any renderer call
5. Add structured audit logging for allow, deny, partial, and escalate
